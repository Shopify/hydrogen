# Tech Debt

Tracked items that are known shortcomings, deferred decisions, or missing guardrails. Each entry should describe what's missing, why it matters, and what "done" looks like.

## Shop Pay button analytics globals are not supplied by Hydrogen

**Status:** Open
**Added:** 2026-05-29

**What:** The Shop Pay ESM custom element emits `shop_js_ui_impression/1.1` only after reading legacy Shopify analytics attributes from globals such as `window.ShopifyAnalytics`, `window.analytics`, or `window.trekkie`. It asks for fields including `contentLanguage`, `microSessionCount`, `microSessionId`, `referer`, `shopId`, `themeCityHash`, `themeId`, `uniqToken`, and `visitToken`. Hydrogen currently does not create or hydrate those globals for headless storefronts.

**Why it matters:** In plain headless apps and local examples, the custom element can render correctly but have no Trekkie attributes available. The current Shop JS analytics path appears to skip producing the impression event when those attributes are empty, so Shop Pay button impressions may be missing or under-attributed outside environments that already load Shopify analytics globals.

**Done when:** Hydrogen either supplies the required analytics context/globals in headless environments, or the Shop Pay custom element accepts explicit tracking/analytics inputs that Hydrogen can pass without relying on global Trekkie state.

## Shop Pay button cannot accept custom checkout search params

**Status:** Open
**Added:** 2026-05-29

**What:** The Shop Pay ESM custom element constructs checkout URLs internally and only appends its supported params: `payment`, `source`, `source_token`, and `channel`. It does not preserve arbitrary query params from `store-url`, and it does not expose an API for callers to pass additional search params. In headless, checkout URLs often need tracking/session query params so checkout can keep the buyer tied to the same storefront analytics session.

**Current workaround:** Hydrogen points the Shop Pay custom element at the storefront app origin and handles the resulting same-origin checkout routes. In production, `/checkout` redirects to the current Storefront API cart checkout URL, while `/cart/{variant}:{quantity}` redirects to the Shopify store-domain cart permalink. When a cart exists, the cart permalink interceptor also merges search params from the current cart checkout URL so tracking/session params are preserved before adding the Shop Pay params generated by the custom element. In local development, the React and Vue development bundles attach a capture-phase click handler that builds the same-origin `/checkout` or `/cart/...` URL directly so localhost keeps its protocol and port.

**Why it matters:** Hydrogen has to route Shop Pay traffic through checkout interceptors to preserve headless tracking/session params that the custom element cannot accept directly. Local development still needs a framework-specific click workaround because the custom element builds `https://{hostname}/checkout` or `https://{hostname}/cart/...` from `store-url`, dropping the port and forcing HTTPS for localhost URLs.

**Done when:** The Shop Pay custom element supports caller-provided search params, preserves needed tracking/session params when building both checkout and variant cart URLs, and supports localhost origins without dropping the port or forcing HTTPS.

## Hydrogen error logging policy

**Status:** Open
**Added:** 2026-05-25

**What:** Runtime helpers currently log errors ad hoc. Shop Pay script loading logs a console error on failure, but there is no package-level policy for when to log, throw, expose reactive error state, or invoke a user-supplied reporter.

**Why it matters:** Silent runtime failures are difficult to debug, but unstructured console logging can be noisy and inconsistent across framework adapters.

**Done when:** Hydrogen has a documented error reporting contract for browser runtime failures, including adapter behavior, optional user hooks, and test expectations.

## No cross-entrypoint import boundary enforcement

**Status:** Open
**Added:** 2026-05-05

**What:** There is no automated check preventing framework adapter entrypoints from importing runtime code that should stay out of those adapter bundles. Entrypoints are meant to be independently tree-shakeable, but nothing enforces this today.

**Why it matters:** A single cross-entrypoint import silently couples entrypoints at build time — consumers who only import a framework adapter could end up pulling in unrelated code, breaking tree-shaking guarantees.

**Ideal solution:** `eslint-plugin-import`'s `no-restricted-paths` rule, which enforces path-based boundaries at lint time (editor feedback). However, oxlint — our target linter — [does not yet support this rule](https://github.com/oxc-project/oxc/issues/13789).

**Acceptable interim solutions (pick one):**
1. A post-build boundary check script (~30 lines) that walks each entrypoint's built output and asserts no imports cross boundaries. Runs in CI.
2. Run ESLint alongside oxlint for this single rule, using `eslint-plugin-oxlint` to disable overlap.

**Done when:** A CI check fails if an adapter entrypoint imports code outside its intended dependency boundary.

## Per-query GraphQL API version override

**Status:** Open — deferred, low priority
**Added:** 2026-05-07

**What:** The Storefront API version is currently set once at client creation time (`DEFAULT_API_VERSION`). There is no mechanism to override the version for an individual query.

**Use case:** Developers sometimes need to hit `unstable` or a newer API version for specific queries before doing a full migration. frandiox raised this as a known Hydrogen use case.

**Why it's deferred:** Overriding the version per-query is a type-safety footgun — `gql.tada` infers types against the schema of the configured version. A query running against a different version would have types that don't match the actual response shape, with no compiler warning.

**Possible approaches:**
1. A `dangerously_overrideVersion` parameter on `graphql()` — makes the risk explicit in the API name.
2. Instruct users to create a separate `createStorefrontClient({ config: { apiVersion } })` instance for unstable queries, keeping type boundaries clean.

**Done when:** We decide on an approach and either implement the escape hatch or document the separate-client workaround.
